# Test Cases — Cart Operations & Product Search/Filter/Sort

Reference SUTs: [saucedemo.com](https://www.saucedemo.com/),
[automationexercise.com](https://automationexercise.com/). Substitute the
actual project SUT and its real test data/URLs when reusing this suite.

| ID | Title | Preconditions | Steps | Test Data | Expected Result | Priority | Type |
|---|---|---|---|---|---|---|---|
| TC-CART-001 | Add item to cart updates cart badge/count | User is on a product listing or product detail page; cart is currently empty | 1. Note current cart badge count (0). 2. Click "Add to Cart" on a product. 3. Observe the cart icon/badge. | Any in-stock product | Cart badge count increments by the quantity added (e.g. 0 → 1); a confirmation toast/message is shown (if the UI provides one); item appears in the cart when opened | High | Functional |
| TC-CART-002 | Remove item from cart updates cart badge/count and total | Cart contains at least 2 distinct items | 1. Open the cart. 2. Click "Remove" on one item. 3. Observe the cart badge and order summary. | Cart with Item A (qty 1) and Item B (qty 1) | Removed item disappears from the cart list; cart badge decrements accordingly; subtotal/total recalculates to reflect only the remaining item(s) | High | Functional |
| TC-CART-003 | Update item quantity recalculates line total and cart total | Cart contains one item with a known unit price | 1. Open the cart. 2. Change the quantity field for the item from 1 to 3 (via input or +/- stepper). 3. Confirm/blur the field if required. | Item unit price = 12.50, qty changed 1 → 3 | Line total updates to 37.50; cart subtotal/grand total updates to include the new line total; quantity field displays 3 after the update | High | Functional |
| TC-CART-004 | Quantity cannot be set to zero or negative via direct input | Cart contains one item | 1. Open the cart. 2. In the quantity input, attempt to enter `0` or `-1`. 3. Attempt to confirm/blur the field. | Quantity input = `0`, then `-1` | Input is rejected or clamped back to the minimum valid value (typically 1); if zero is submitted via a decrement control, the item is either removed with a confirmation or blocked, but the cart never shows a `0` or negative quantity line item | Med | Boundary |
| TC-CART-005 | Cart contents persist across page navigation within the same session | Cart contains at least one item | 1. Add an item to the cart. 2. Navigate away to a different page (e.g. home page, another category). 3. Return to the cart. | Any in-stock product | Cart still contains the previously added item with the correct quantity; cart badge count is unchanged by the navigation | Med | Functional |
| TC-CART-006 | Cart contents persist after browser refresh | Cart contains at least one item | 1. Add an item to the cart. 2. Refresh the browser page (F5 / reload). 3. Open the cart. | Any in-stock product | Cart still contains the item(s) added before the refresh, with correct quantity and price (assuming the store's cart persistence model — session/local storage/account — is expected to survive a refresh) | Med | Functional |
| TC-SRCH-001 | Product search returns relevant results for a valid keyword | User is on the storefront home/search page | 1. Enter a known product keyword into the search box (e.g. "dress" or "shirt" on Automation Exercise). 2. Submit the search. | Search term = `dress` | Results page shows only products whose name/description reasonably matches the keyword; result count is displayed if the UI supports it; no unrelated products appear at the top of results | Med | Functional |
| TC-SRCH-002 | Search with no matching results shows an empty-state message | User is on the storefront home/search page | 1. Enter a keyword guaranteed not to match any product (e.g. a random string). 2. Submit the search. | Search term = `zzqxnotarealproduct123` | A clear "No products found" (or equivalent) message is shown; no error page/crash occurs; suggested actions (e.g. "browse categories") may be shown if the UI supports it | Med | Negative |
| TC-SRCH-003 | Sorting product listing by price (low to high) orders results correctly | User is viewing a product category/listing page with 3+ products of differing prices | 1. Select the "Price: Low to High" sort option from the sort control. 2. Observe the resulting order of products. | Category with products priced e.g. 9.99, 15.00, 29.99, 45.00 | Products are displayed in strictly ascending price order from left-to-right / top-to-bottom per the store's grid layout; sort selection persists on pagination (if applicable) | Med | Boundary |
